# 小白AI产品记忆

最后更新：2026-05-13

## 一句话定位

小白AI不是单纯的 AI 工具导航，而是面向 AI 新手的“实战学习与任务推进系统”：用户进站后先说自己想做成什么事，小白把目标拆成可完成、可检查、可复盘的 0-1 任务，并记住用户上次做到哪里。

## 目标用户

- 完全不会 AI，但愿意开始尝试的普通用户。
- 想用 AI 提高办公、内容、学习、编程、自动化效率的人。
- 看到工具很多但不知道怎么选、怎么开始的人。
- 想跟着真实案例完成一个可交付成果的人。
- 国内用户，对 DeepSeek、Kimi、Claude Code、Codex、Dify、Coze、n8n、Cursor 等工具和模型有实际使用需求。

## 产品护城河

不要只堆工具列表。别人很容易复制工具目录，但不容易复制下面这些东西：

- 用户进度：小白记住用户上次做到了哪个阶段，提醒下一步。
- 实战任务：按真实目标组织任务，而不是按工具名堆教程。
- 复盘资产：用户完成任务后发社区复盘，沉淀真实案例。
- 解读能力：资讯不只是搬运标题，而是解释“对新手有什么用、该做什么任务”。
- 分类可信：明确区分 Agent、模型、应用平台、工作流工具。
- 本土适配：关注国内可用路径、国内模型 API、中转站、支付、网络、部署和安全边界。

## 核心产品结构

### 学习主线

学习系统当前采用“大科目 -> 小科目 -> 教程/任务”的结构。用户每次更新学习内容时，必须同步更新路线图和科目归属，不能再回到单页堆教程。

学习应该从低级到高级贯通：

1. L0：认识 AI，不迷信，不乱填隐私。
2. L1：每天会用 AI 做真实小事，会提问、给材料、要格式、验收和修复；优先覆盖写作改稿、读资料、辅助决策、表格数据整理、生活管理、学习陪练这六类普通人最高频能力。
3. L2：把基础能力组合到真实场景，比如职场一天提效、学生学习提升、家庭管理、副业内容输出和一人公司起步。
4. L3：形成自己的常用模板和上下文材料包。
5. L4：搭建一个简单 Agent / 知识库 / 自动化流程。
6. L5：理解模型、API、Token、价格、上下文、隐私。
7. L6：让工程 Agent 或自动化工具参与真实项目。

学习不是目的，产出才是目的。每个阶段完成后最好有一点奖励，例如 XP、徽章、复盘入口、下一阶段建议。

学习内容必须同时覆盖个人线和团队线：

- 个人线要包含一人公司、自由职业、个人创作者、居家副业、职场成长、学生学习、家庭生活管理、亲子教育、个人财务、健康习惯、旅行规划、个人 Agent 管家等具体场景。
- 团队线要覆盖企业知识库、客服、销售、SOP、自动化、团队 AI 使用规范、Agent 上线验收等具体场景。
- 行业落地不能少于 20 个行业，每个行业都要写清楚提效、赋能或变现方式，不要只写行业名称。
- 学习页面面向用户展示“能做什么、会带来什么变化、下一步点哪里”，不要展示内部规则解释，例如“不要堆教程”“每个小科目只放相关内容”这类话。
- 学习内容要有图示表达。优先用“输入 -> 学习/AI处理 -> 结果/交付物”的流程图，让普通用户不只靠读文字理解。
- 网站要轻、整洁。旧学习页、重复路线图、未引用的发布包、临时 docx 脚本和不进入网站体验的内容应清理掉，避免和正式学习系统并存。
- 每次站长提供飞书知识库、AI 文章、研报、视频、资讯或外部教程时，必须优先阅读正文。目录只用于判断分类，正文才用于抽取知识点。更新流程是：记录来源和正文范围 -> 抽概念/流程/工具/案例/风险 -> 映射到小白AI大科目和小科目 -> 转成 MVP 教程和任务 -> 检查路线图是否同步。不要只复制别人的目录、标题或分类。

### 实战任务系统

已建立 `/missions` 任务库和 `/start` 任务继续台。

当前任务线包括：

- `claude-code-deepseek-project`：用 Claude Code 接 DeepSeek V4 改一个真实项目。
- `codex-small-feature`：用 Codex 完成一个网页小功能。
- `kimi-k26-long-doc`：用 Kimi K2.6 分析长文档并生成行动清单。
- `ai-daily-use-bootcamp`：用 5 个真实小任务练会提问、给材料、要格式、验收、修复和保存模板。
- `ai-writing-rewrite-drill`：用真实消息、邮件、通知或内容草稿练会 AI 写作和改稿。
- `ai-reading-material-drill`：把文章、截图、聊天记录或说明书拆成事实、风险、问题和行动清单。
- `ai-decision-helper-drill`：让 AI 辅助整理选项、标准、风险和低成本验证动作，但保留人工拍板。
- `ai-table-data-cleanup-drill`：把账单、名单、订单或打卡记录整理成表格、异常和待确认清单。
- `ai-life-management-drill`：把家庭、旅行、亲子、健康、财务或日程拆成 7 天可执行生活管理卡。
- `ai-learning-coach-drill`：让 AI 陪练一个知识点，完成讲解、出题、错题解释和 3 天复习表。
- `workday-ai-boost-plan`：把职场一天拆成资料阅读、沟通写作、表格整理和下班复盘，形成明天可用的提效卡。
- `student-ai-study-upgrade-plan`：把一个知识点拆成讲解、练习、错题和 3 天复习表，避免只听懂不掌握。
- `family-ai-household-plan`：把家庭日程、孩子学习、旅行、购物、开支或家务整理成 7 天协作卡。
- `side-income-content-sprint`：把技能、经历、服务或产品变成 7 天内容输出计划和第一条可发布样稿。
- `one-person-company-starter-plan`：把定位、服务包、报价、内容、咨询话术和交付验收压成 7 天起步包。
- `dify-knowledge-base-bot`：用 Dify 搭一个客服知识库。
- `n8n-ai-news-automation`：用 n8n 做每日 AI 资讯自动化。
- `xiaohongshu-ai-content-loop`：做一条小红书 AI 内容流水线。

进度 v1 使用浏览器 `localStorage`：

```text
xiaobaiai:mission-progress:v1
```

用户完成全部步骤后，可以通过 `/api/growth/xp` 领取对应任务 XP。

### 小白提醒机制

用户回到 `/start` 时，小白应该提醒：

- 当前任务是什么。
- 已完成几步。
- 下一步应该做什么。
- 完成后应该发复盘，把经验沉淀下来。

这条机制非常重要，是从“工具站”转向“陪伴式任务系统”的关键。

## 工具和模型分类规则

这部分必须保持准确，用户已经明确指出之前排名和分类不靠谱。

### 页面归属和入口规则

- 顶层导航只保留真正的大入口：开始学习、工具资源、实战展示、AI资讯。社区、成长舱、关于我们放在更多入口。
- 模型不是顶层导航主入口，归到“工具资源”和学习路线里的“模型选择”。模型页负责帮用户选大脑：场景、价格、上下文、API、本地模型和隐私边界。
- Agent 安装不是顶层导航主入口，归到“工具资源”和学习路线里的“Agent 安装”。安装页负责装执行工具，教程页负责带用户做项目。
- 工具资源页不是工具海报墙，顶部必须先放清晰整齐的资源入口：模型选择、Agent 安装、工具分类、学习路线。分类列表放在资源入口后面。
- 教程、任务、工具、模型、案例的入口要位置合理、视觉整齐。不要让多个相同层级的入口散落在导航、页面角落和重复卡片里。

### Agent / 工程 Agent

属于能读项目、执行任务、改代码、跑命令、持续迭代的工具：

- Codex
- Claude Code
- Hermes
- Cursor Agent
- GitHub Copilot coding agent 类能力

排序应按实际执行能力、热度、可靠性、国内可用性综合判断。不要让不知名或不主流工具长期排最前。

### 模型 / 模型 API

属于可被工具调用的模型后端，不要写成 Agent：

- DeepSeek V4
- Kimi K2.6
- GPT 系列
- Claude 系列
- Gemini 系列
- Qwen 系列

DeepSeek V4 只是模型或模型 API 后端，不是 Agent。可以出“Claude Code 接 DeepSeek V4 API”教程，但标题和分类必须清楚。

### 应用搭建 / 知识库 / Agent 平台

属于用来搭应用、知识库、Bot、流程的产品：

- Dify
- Coze
- FastGPT
- Langflow

### 工作流自动化

属于自动触发、连接多个系统、定时运行、消息通知的工具：

- Make

## 内容策略

### 要做的内容

- 当前最热、用户真会搜的教程：
  - Claude Code 接 DeepSeek V4 / 国内模型 API。
  - Claude Code 国内中转站配置和风险提醒。
  - Codex 国内使用与真实项目改代码。
  - Kimi K2.6 长文档和多模态使用。
  - Dify 知识库切分、召回、转人工边界。
  - n8n AI 资讯、日报、通知自动化。
- 每篇教程都要落到一个可执行任务，最好能接入 `/missions`。
- 教程至少要做到“落地 MVP”的程度：用户跟完以后必须能交付一个最小可用成果，例如一个可运行流程、一个可复用模板、一个可演示 Bot、一个自动化任务、一个行业小方案或一个可展示页面。
- MVP 教程必须包含：目标场景、准备条件、操作步骤、示例输入、可见结果、验收标准、常见失败修复、下一步迭代和对应实战任务入口。
- 不要把教程写成单纯介绍或工具说明。模型页可以做选型，工具页可以做入口，真正教程要把用户带到一个能落地的成果。
- 资讯内容要变成“小白解读”，不要只搬新闻：
  - 发生了什么。
  - 对新手有什么影响。
  - 是否值得立刻学。
  - 可以用它完成什么 0-1 任务。
  - 相关工具、教程、任务入口。
- AI 资讯必须有对应内容图。新闻本身有产品截图、发布会图、模型榜单图、论文图、界面图、官网图或演示图时，优先使用与这条新闻内容直接相关的图片。
- 更新 AI 资讯时，如果没有现成配图但有来源 URL，要优先截取来源页面或产品页面截图，放到资讯列表和详情页；不要默认用小白头像、来源 logo 或抽象图代替真实新闻画面。
- 只有在来源无图、截图不可用、或版权/访问限制导致不能使用时，才允许退回来源 logo 或系统生成的兜底视觉。
- 外部知识库内容要变成“小白吸收”：读正文、提炼方法、判断适合个人还是团队、说明能带来什么提效或变现结果，再落到教程、任务、复盘模板和下一步入口。不能把小白AI做成外部知识库索引站。
- 提示词、上下文工程、AI 搜索和研究判断是学习系统的底层能力。新增外部知识源、资讯、研报、论文、产品库或教程时，优先检查这些内容是否需要进入“提示词与上下文工程”“AI 搜索与研究判断”“AI 知识源正文拆解”，再决定是否新增行业、工具或前沿分支。
- 站长明确要求优先做“教人使用 AI”。后续学习系统扩展时，优先补普通人每天怎么用 AI、怎么完成真实小任务、怎么检查和修复结果、怎么沉淀模板。企业服务、社区增长、前沿研究和商业化内容可以做，但不能压过基础使用训练。

### 不要做的内容

- 不要填过时工具。
- 不要假装测试过没测试过的产品。
- 不要把模型说成 Agent。
- 不要把教程写成泛泛而谈的百科。
- 不要只做工具选择器，用户真正要的是“我现在想做什么事，下一步怎么做”。

## 社区策略

社区应该服务于“复盘”和“案例沉淀”，而不是泛灌水。

优先内容：

- 我用某工具完成了什么。
- 我卡在哪里，怎么解决。
- 我的提示词、配置、流程。
- 我的失败记录。
- 新手避坑。

每个任务完成后都应该引导用户发复盘。复盘不是炫耀，是让用户把经验变成资产。

## 近期产品方向

优先级建议：

1. 稳定资讯抓取和资讯状态可观测。
2. 补一批真实、当前热门的教程和任务，尤其 Claude Code、Codex、DeepSeek V4、Kimi K2.6、Dify、n8n。
3. 让任务系统和学习主线更强绑定：学完一个阶段，立刻推荐一个可完成任务。
4. 增加“上次做到哪里”的更强提醒，未来可以从 localStorage 升级到账号级同步。
5. 建立“候选资讯池 + 人工确认 + 小白解读”机制，减少假新闻和低质量内容。

学习页结构规则：

- 小科目页必须明确展示“先学教程 -> 做任务验收 -> 写复盘沉淀”的行动路径。
- 教程详情页底部必须有统一下一步行动区，至少包含实操任务、下一课或回小科目、复盘入口、当前交付物提醒。
- 教程和任务不能混成一个普通列表：教程负责讲方法和交付物，任务负责让用户按步骤做出可检查 MVP。
- 每次新增或调整教程、任务、小科目时，要同时检查小科目行动路径、教程底部行动区、路线图节点是否仍然能闭环。
- 当用户要求“加深知识”时，优先补教程里的底层原理、判断标准、边界误区、迁移方法，不要默认继续新增任务。
- 知识深度要让用户知道为什么这样做、什么时候不该这样做、怎么迁移到别的场景；不是只把步骤写长。
- 知识深度层要覆盖大科目共性：办公提效讲读者、事实、责任和行动；内容创作讲受众、真实素材、渠道、版权和复盘；个人成长讲真实问题、7 天目标、人工确认和可见变化。

## 产品口径

小白AI的语气应该是：

- 直接、诚实、接地气。
- 不吓唬新手，也不神化 AI。
- 鼓励用户先做一个小环节，从 0 到 1。
- 遇到工具热度、排名、分类，要基于当前事实谨慎更新。

一句核心话术：

> 不是先问“该用哪个工具”，而是先问“你现在最想做成什么事”。完整项目太大，我们就先做好其中一个环节，这就是开始。

## 站长协作习惯

站长更喜欢“直接一起做”，不喜欢长时间停留在建议层。除非有高风险或信息缺失，否则应该直接改代码、跑验证、提交并给部署命令。

沟通习惯：

- 用中文回答，直接、实用、少绕。
- 给命令时要能直接复制执行。
- 用户经常把服务器终端输出贴回来，要基于输出判断下一步，不要重复讲大段理论。
- 用户会同时开多个会话让不同助手改项目；如果他说“另一个会话改好了”，默认要告诉他先确认是否 push，然后服务器 `git pull` 一次即可拉到所有已 push 的提交。
- 用户问“上一版没跑，这次一起跑吗”，回答重点是：只要都已 push，一次 `git pull` 会连续拉到最新提交，然后一次 build/restart 即可。
- 生产服务器上有本地生成文件时，不要建议粗暴 `git reset --hard`。先 `git status --short`，必要时用 `git stash push -u -m "server local generated files before pulling latest"`。
- 用户重视上线验证。部署命令后要附上 `curl -I` 和 `pm2 logs ... --nostream`。
- 用户希望我们记住项目状态、部署路径、PM2 名称、Supabase SQL 是否需要执行、cron 是否需要检查。

做完代码后默认流程：

1. 本地跑 `npm.cmd run typecheck`。
2. 本地跑 `npm.cmd run lint`。
3. 重要 UI/路由改动跑 `npm.cmd run build`。
4. `git diff --check`。
5. commit + push。
6. final 里告诉用户 commit hash、改了什么、服务器怎么拉。

服务器部署口径：

```bash
cd /var/www/xiaobaiai
git status --short
git pull
git log --oneline -n 5
npm run build
pm2 restart xiaobaiai --update-env
pm2 flush xiaobaiai
curl -I https://www.xiaobaiai.cn/
pm2 logs xiaobaiai --lines 80 --nostream
```

如果涉及新增页面或任务，要在部署命令里加具体 URL 验证，例如：

```bash
curl -I https://www.xiaobaiai.cn/missions/ai-ppt-first-deck
curl -I https://www.xiaobaiai.cn/growth
```
